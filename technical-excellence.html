Section 1

Code Quality and Conventions

Consistency is the foundation of velocity. When every file follows the same patterns, every API uses the same conventions, and every engineer can read any part of the codebase without friction — the team moves fast because cognitive overhead is low. Your job as a leader is to establish these conventions, document them, and make them the path of least resistance.

Key Insight

The goal of conventions is not uniformity for its own sake. It is to eliminate the decisions that do not matter (naming style, file structure, import order) so the team can focus on the decisions that do (architecture, design, trade-offs). Every convention that is automated is a debate that never happens again.

The Conventions Checklist — What Every Team Needs Documented

  1. Coding standards — Naming conventions, file/package structure, error handling patterns, logging standards. Enforced by linters and formatters, not humans.
  2. API conventions — URL structure, versioning strategy, error response format, pagination approach, authentication patterns. Document once, follow everywhere.
  3. Database conventions — Naming (tables, columns, indexes), migration practices, indexing strategy, query patterns to use and avoid.
  4. Package/module structure — Where new code goes, how features are organized, layer boundaries (controllers, services, repositories, domain). A new team member should find any file within 30 seconds.
  5. README and documentation standards — Every repo has a README with: what the service does, how to run it locally, how to run tests, architecture overview, key decisions. If it is not in the README, it does not exist.
  6. Language feature standards — Which modern language features the team has adopted, which patterns to prefer (immutability, data classes, async patterns), and which legacy patterns to migrate away from when touching old code.
Common Pitfall

Conventions that live in someone's head. "Everyone knows we do it this way" — until a new person joins and does it differently, and nobody can point to a written standard because one does not exist.

Better Approach

A living conventions document in the repo, linked from the README, referenced in MR reviews. Updated quarterly. Every convention includes the "why" — not just the "what". When people understand the reasoning, they follow the convention even in edge cases the document did not anticipate.

💡

Start with the conventions that cause the most friction in code reviews. If every other MR has a comment about naming or file structure, that is the convention to document first. Automate what you can (formatters, linters), document the rest, and reference the document in reviews.

Section 2

The Developer Experience

How quickly can a new developer go from cloning the repo to running the application locally? If the answer is more than 30 minutes, your developer experience needs work. Every minute of friction in setup, builds, and local testing is a minute lost from productive work — multiplied by every developer, every day.

Local Setup

One-command setup is the gold standard. A script (or Makefile, or docker-compose) that installs dependencies, seeds the database, and starts the application. If setting up requires reading a 20-step wiki page, people will skip steps and waste hours debugging environment issues.

Pre-commit Hooks

Formatting, linting, and basic checks run automatically before every commit. This catches 80% of code review nits before the MR is even created. Use tools like Husky, pre-commit, or Git hooks natively. The rule: if a human has to check it, automate it.

Auto-formatting

Configure a formatter (Prettier, Black, Spotless, gofmt, ktlint — whatever fits your stack) and enforce it in CI. Formatting discussions in MRs are the lowest-value use of engineering time. Automate them away completely.

Fast Builds

A build that takes 10 minutes is a build that breaks flow. Profile your build. Cache aggressively. Parallelise where possible. Set a build time budget (e.g., under 3 minutes for local, under 8 for CI) and treat regressions as bugs.

Compiler Warnings

Treat warnings as errors. A codebase with 500 warnings teaches engineers to ignore warnings — including the important ones. Start by fixing existing warnings in a dedicated cleanup sprint, then enforce zero-warning builds in CI.

Scripts and Tooling

Common tasks should be scripted: running tests, generating test data, seeding environments, deploying to staging, running database migrations. If someone has to look up how to do a common task, write a script for it. If a script exists but is undocumented, add it to the README.

The New Joiner Test

Ask someone who just joined the team: "How long did it take you to go from clone to running locally?" and "What was confusing?" Their answers reveal every friction point your setup has. Run this test with every new team member and fix what they find. The best developer experience documentation is written by the person who just went through it for the first time.

⚠

Developer experience degrades silently. Everyone adapts to the friction and stops noticing it. Run a quarterly audit: how long is the build? How many steps to set up locally? How many flaky tests are there? If the numbers are getting worse, invest before the team normalises the pain.

Section 3

Code Reviews as Culture

A code review is the most frequent teaching moment a leader has. Done well, reviews raise the quality bar, spread knowledge, and build shared ownership of the codebase. Done poorly, they become rubber stamps or gatekeeping rituals that slow everyone down.

Key Insight

An MR review is the most frequent teaching opportunity you have. Every review is a chance to raise someone's understanding — or a missed opportunity.

The MR Review Checklist

What reviewers should check:

  1. Domain correctness — Does the implementation actually solve the problem described in the ticket? Are business rules correctly applied?
  2. Design — Is the code well-structured? Single responsibility? Right level of abstraction? Could a new team member understand this without context?
  3. Naming — Do class, method, and variable names clearly communicate intent?
  4. Error handling — Are exceptions handled intentionally? Are error messages helpful for debugging? Are edge cases considered?
  5. Tests — Are tests present, meaningful, and covering the important scenarios? Would they catch a regression?
  6. Observability — Does the change include appropriate logging and metrics? Would you be able to debug this in production?
  7. Security — Input validation, parameterized queries, authorization checks, no exposed secrets?
  8. Conventions — Does the code follow the team's documented standards?
Weak MR Comment

"This is wrong. Use X." — Tells the author what to do but not why. Feels like a command, not coaching.

Strong MR Comment

"This method updates two tables but is not wrapped in a transaction. If the second write fails, you will have inconsistent data. Consider wrapping in a transaction. Here is a resource that explains the pitfalls." — Explains the risk, suggests the fix, and teaches. The author walks away understanding why.

MR Process Standards

  • Turnaround time: MRs should be reviewed within 4 hours during working hours. Stale MRs kill velocity and morale.
  • Size: Keep MRs under 400 lines of meaningful change. Large MRs get rubber-stamped because reviewing 2000 lines is exhausting. If a feature requires more, break it into a chain of smaller MRs.
  • Self-review: Authors should self-review the diff before requesting review. This catches obvious issues and shows respect for the reviewer's time.
  • Description: Every MR should explain what it does, why it does it, and how to test it. A diff without context is a puzzle, not a review.
💡

The quality of your MR reviews sets the team's quality bar. If you rubber-stamp MRs, the team learns that quality is optional. If you give thoughtful, teaching-oriented feedback, the team learns that quality matters. Be the reviewer you want everyone to become.

Section 4

Testing Strategy

If bugs consistently reach production, the answer is not "be more careful" — it is "fix the system." Testing is a strategy, not a chore. Your job as a leader is to build a testing culture where the right things are tested at the right level, and the test suite gives the team genuine confidence to deploy.

Key Insight

Coverage numbers tell you what code was executed. They do not tell you whether your tests would catch a real bug. A test that checks "does not throw an exception" has coverage but no value. Focus on test quality, not test quantity.

The Testing Pyramid, Applied

  1. Unit tests (the base — fast, many): Test behaviour, not implementation. A good unit test describes what the code should do, not how it does it. If refactoring internals breaks your tests, they are coupled too tightly.
  2. Integration tests (the middle — real dependencies): Use real databases (via Testcontainers or similar), not in-memory substitutes. The whole point of integration tests is verifying that your code works with the real thing. Fake databases hide real bugs.
  3. Contract tests (API boundaries): When services communicate, contract tests verify that producers and consumers agree on the shape of data. This catches breaking changes before they reach a shared environment.
  4. E2E tests (the top — critical paths only): Expensive to write and maintain. Reserve for the most critical user journeys. Keep them fast by testing happy paths and the most important failure modes, not every permutation.

Test Data Strategy

Tests need data. If every test builds its own data from scratch, tests become slow and brittle. Invest in test data builders, factories, or fixtures. Make it easy to create realistic test data with one line of code. Shared test utilities are infrastructure worth maintaining.

Left-Shift QA

Move quality checks earlier in the pipeline. Do not wait for a "QA phase" after development. Developers own testing. Acceptance criteria include test expectations. MR reviews check test quality. The earlier a bug is caught, the cheaper it is to fix — by orders of magnitude.

Flaky Test Zero Tolerance

A flaky test is worse than no test. It erodes trust in the entire suite. When the team starts ignoring test failures because "it is probably just that flaky test," you have lost the safety net. Fix or remove flaky tests the day they appear.

Mutation Testing

Code coverage tells you what code was executed. Mutation testing tells you whether your tests actually catch mistakes. Tools like PIT, Stryker, or mutmut modify your code in small ways and check if your tests fail. If they do not, the tests are providing false confidence.

Section 5

CI/CD and Deployment Practices

Your CI/CD pipeline is the heartbeat of your engineering practice. A fast, reliable pipeline means the team deploys with confidence. A slow, flaky pipeline means the team avoids deploying — and problems compound.

Key Insight

The time from "code is merged" to "code is in production" is one of the most revealing metrics of engineering health. If a one-line fix takes 3 days to reach production, your pipeline has friction that needs attention. Elite teams deploy on demand. Low-performing teams deploy monthly. The pipeline is often the difference.

The CI/CD Maturity Checklist

  1. Automated builds — Every commit triggers a build. No manual steps. No "works on my machine."
  2. Automated tests in pipeline — Unit, integration, and contract tests run on every MR. If tests pass, the code is deployable.
  3. Pipeline stays green — A broken pipeline is treated as a blocking issue. The person who broke it owns fixing it immediately.
  4. Fast feedback — The pipeline gives a pass/fail result within 10 minutes. Anything longer erodes the feedback loop. Profile, cache, parallelise.
  5. Automated deployment to staging — Merging to main deploys to staging automatically. Manual staging deployments create bottlenecks.
  6. One-click production deploy — Production deployments are automated and repeatable. No SSH, no manual scripts, no "only Dave knows how to deploy."
  7. Rollback capability — If something goes wrong, you can roll back in under 5 minutes. Deployments without rollback plans are gambles.
  8. Environment parity — Staging mirrors production as closely as possible. Bugs that only appear in production because staging is different are entirely preventable.

Shorter Feedback Cycles

The faster a developer learns their change is broken, the cheaper it is to fix. Aim for: pre-commit hooks catch formatting in seconds, CI catches build errors in minutes, integration tests catch logic errors in under 10 minutes, production monitoring catches escaped bugs in minutes. Each layer is a faster feedback loop.

Efficient Pipelines

Profile your pipeline like you profile your application. Where is the time going? Common wins: parallelise independent test suites, cache dependencies between builds, use incremental compilation, skip unchanged modules. A 15-minute pipeline reduced to 5 minutes saves the team hours every day.

Containerisation

Containers ensure that what works locally works in CI works in production. Define your runtime in a Dockerfile, compose your dependencies, and eliminate "it works on my machine." Containerised builds are reproducible, portable, and cacheable.

Feature Flags

Decouple deployment from release. Deploy code to production behind a flag. Enable it gradually (canary, percentage rollout, specific users). If something goes wrong, disable the flag — no rollback needed. Feature flags turn deployments from high-stakes events into routine operations.

💡

Measure your deploy frequency and lead time monthly. These are two of the four DORA metrics and they directly predict team health. If the numbers are not improving, investigate what is slowing the pipeline down.

Section 6

Architecture Principles

Architecture is not a document written once and filed away. It is a set of living principles that guide every technical decision the team makes. Your job as a leader is to ensure these principles are understood, applied consistently, and revisited as context changes.

Principles Every Service Should Follow

  1. Externalise configuration — No hard-coded URLs, credentials, or environment-specific values in code. Configuration lives in environment variables or config services. The same artifact deploys to every environment.
  2. Stateless services — Application instances should hold no local state between requests. Session data, caches, and temporary files go in external stores. This enables horizontal scaling, zero-downtime deployments, and simple recovery.
  3. Single Responsibility — Each service, class, and method owns one cohesive concern. When a service does too many things, it becomes hard to test, hard to change, and hard to understand. If you cannot describe a service's purpose in one sentence, it is doing too much.
  4. Design for failure — Every external dependency will fail. Every network call will timeout. Every database will go down. Build circuit breakers, retries with backoff, graceful degradation, and fallback behaviour. The question is not "will it fail?" but "how will it behave when it does?"
  5. API-first design — Define the contract before building the implementation. Use API specs (OpenAPI, GraphQL schemas) as the source of truth. This enables parallel development (frontend and backend work simultaneously), contract testing, and documentation that stays in sync.
  6. Infrastructure as code — All infrastructure is defined in code and version-controlled. If it is not in the repo, it does not exist. No manual console changes that cannot be reproduced or audited.

Cross-Functional Requirements (CFRs)

Performance, scalability, reliability, security, accessibility — these are not "nice to haves" that get addressed before release. They are requirements that should be defined upfront and tested continuously. For every feature, ask: "What is the expected load? What is the acceptable latency? What happens if it fails?" Document these alongside functional requirements.

Microservice Practices

If your team builds microservices: own the full lifecycle (build, deploy, run, monitor). Use consistent patterns across services (error handling, logging format, health checks, config management). Define clear service boundaries based on domain, not technical layers. Keep inter-service communication simple and well-documented.

Database Conventions

Use a migration tool for all schema changes. Every migration is versioned, idempotent, and reviewed. Index based on actual query patterns, not guesswork. Never store secrets in plain text. Use parameterized queries exclusively. Separate read and write models when query patterns diverge.

Design Principles to Model

Teach these in reviews: Tell Don't Ask (let objects own their behaviour), Composition over Inheritance (favour "has-a" over "is-a"), Fail Fast (catch errors at the boundary, not three layers deep), Law of Demeter (avoid long method chains through objects), Rich Domain Models (put business logic in entities, not in services).

⚠

Architecture principles are useless if only one person knows them. Document them as Architecture Decision Records (ADRs) in the repo. Each ADR captures: the context, the decision, the alternatives considered, and the consequences. When someone asks "why do we do it this way?" the answer is in the repo, not in someone's memory.

Section 7

Observability and Resilience

Having monitoring tools is not the same as having an observability culture. The goal is this: if something goes wrong in production, the team knows within 5 minutes — from dashboards, not from users. And when it does go wrong, the system degrades gracefully rather than collapsing entirely.

The Observability Essentials

  1. RED metrics for every service — Rate (requests per second), Error (percentage failing), Duration (p50, p95, p99). If you can answer these three questions for every service, you have a solid foundation.
  2. Structured logging — JSON-formatted logs with consistent fields across services. Every log entry includes a correlation/trace ID. Never log sensitive data (passwords, tokens, PII). Log at appropriate levels (INFO for business events, WARN for recoverable issues, ERROR for failures).
  3. Distributed tracing — When a request flows through multiple services, a shared trace ID ties the story together. Without it, debugging distributed issues is guesswork.
  4. Actionable alerts — Every alert links to a runbook. Alerts fire on symptoms (error rate, latency), not causes. Set thresholds based on real baselines. Review alert noise monthly — prune anything that fires without action. If the on-call response to an alert is "ignore it," that alert is noise.
  5. Morning dashboard check — Make it a team habit to glance at key dashboards before standup. Catching anomalies early prevents incidents. Over time, people develop intuition for what "normal" looks like.

Feature = Code + Tests + Monitoring

A feature is not done until it is observable. Every new capability should ship with metrics, logging, and (where appropriate) alerts. Build this into your definition of done.

Safety Nets

Every incident should make the system stronger. After each post-mortem, add at least one safety net: a test that would have caught the bug, a monitor that would have alerted earlier, a process check that would have prevented the deploy, or a code-level defence (input validation, circuit breaker, feature flag).

Resilience Patterns

Circuit breakers (stop calling a failing service), retries with exponential backoff (handle transient failures), bulkheads (isolate failures so they do not cascade), graceful degradation (serve a reduced experience rather than an error page), timeouts (never wait forever for anything).

Chaos Engineering (When Ready)

Once your observability and safety nets are solid, deliberately inject failures (kill a pod, slow a dependency, simulate a network partition) to verify that the system behaves as expected. This is not day-one work — it is a maturity milestone. But it is the only way to truly know your system is resilient.

Section 8

Security as a Practice

Security is not a feature you add at the end — it is a mindset you build into the team from the start. Left-shift security means catching vulnerabilities during development, not in a pre-release audit. As a leader, you set the bar.

Security Practices for Every Team

  1. Threat modelling for new features — Before building, ask: "What could an attacker do with this? What data are we exposing? What are the trust boundaries?" A 30-minute session during design prevents weeks of remediation.
  2. Dependency scanning in CI — Automated tools (Snyk, Dependabot, Trivy) flag known vulnerabilities. Make it a merge blocker for critical/high findings.
  3. OWASP Top 10 awareness — Every engineer should know the common vulnerabilities (injection, broken auth, XSS). Run a brown-bag session. Make it part of onboarding.
  4. Secrets management — No secrets in code, no secrets in repos, no shared credentials. Use a secrets manager. Rotate on a schedule. Audit access regularly.
  5. Security in code reviews — Add to the MR review habit: input validation, parameterized queries, output encoding, authentication checks, authorization boundaries.

Security Smells in Code Reviews

Injection

String concatenation in queries instead of parameterized queries.

Missing Input Validation

User input flowing directly into business logic without sanitization.

Exposed Secrets

API keys, passwords, or tokens in code, config, or logs.

Broken Authorization

Endpoints that check authentication but not authorization.

Excessive Data Exposure

API responses returning more fields than the consumer needs.

Missing Rate Limiting

Public endpoints with no throttling.

⚠

If your team discovers a security vulnerability, treat it like a production incident — not a backlog item. Assess the blast radius, patch it, and do a post-mortem. Security debt compounds faster than tech debt.

How to Establish Practices — The Leader's Playbook

  1. Introduce with rationale — Explain the practice and why it matters, connected to real problems the team has experienced.
  2. Document in the repo — If it is not written down, it is not a standard.
  3. Model it yourself — Adopt it in your own code first.
  4. Coach for two weeks — Point out deviations gently, explain the reasoning.
  5. Enforce after the grace period — Reference the documented standard in reviews.

The standard you walk past is the standard you accept.
